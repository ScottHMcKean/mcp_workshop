1. Why MCP

The problem MCP solves

Before MCP, every host re-implemented every integration:

Cursor    ─┐
Claude Desktop ─┼─→ each builds its own GitHub, Slack, DB, ... connectors
your agent ─┘

N hosts × M tools = N × M glue code.
This is the same N×M problem LSP solved for editors and language servers.

After MCP

Genie Code  ─┐
Playground  ─┼─→ MCP ─→ Genie, Vector Search, UC Functions, you.com, your custom App, ...
your notebook ─┘

N + M, not N × M.

USB-C of LLM context. One connector, many devices on either side.

Why it took off (in 18 months)

  • Open spec, open SDKs (TypeScript, Python, Rust, Kotlin, …).
  • Anthropic shipped first; OpenAI, Google, Microsoft followed.
  • Hosts inside Databricks: Genie Code (in-workspace agent, replaced Databricks Assistant in March 2026), AI Playground, your own notebooks. Hosts outside Databricks: Claude Code, Cursor, Claude Desktop, VS Code, ChatGPT desktop, others.
  • Servers: thousands public, plus first-party from Slack, Notion, GitHub, Atlassian, Databricks.
The killer feature is composability. You don't ship one agent — you ship one server, and every host gets it.

What MCP is not

  • Not a runtime. It does not execute your code.
  • Not a model. Provider-agnostic.
  • Not a framework. FastMCP is a framework. MCP is the wire format.
  • Not "function calling." Function calling is one model's API. MCP is a protocol that decouples tools from any model.

What it actually is

A JSON-RPC 2.0 protocol. Two methods you'll see most often:

→ {"method": "tools/list"}
← {"tools": [{"name": "execute_sql", "inputSchema": {...}}, ...]}

→ {"method": "tools/call", "params": {"name": "execute_sql", "arguments": {"query": "SELECT 1"}}}
← {"content": [{"type": "text", "text": "[{\"_c0\": 1}]"}]}

That's the whole wire format. Everything else is convention.

Three primitives a server exposes

People remember tools. The other two are where the leverage is.

Tools — model-controlled

@mcp.tool()
def franchise_sales_trend(franchise: str, days: int = 90) -> list[dict]:
    """30-day per-product sales for a franchise. Use when investigating a slump."""
    ...
  • Model reads your description and decides when to call.
  • Description should say when to call, not what it does.
  • JSON-Schema args, JSON return.

Resources — app-controlled

@mcp.resource("franchise://list")
def franchise_index() -> str:
    return "# Franchises\n- austin\n- portland\n- chicago\n..."

Why it matters: the host can show this in a picker UI. The user attaches it to context. The model never has to read 5,000 franchises into its context window.

Think: file attachments, but addressed by URI and computed on demand.

Prompts — user-controlled

@mcp.prompt()
def diagnose_franchise(franchise: str) -> list[Message]:
    return [user(f"Why is {franchise} underperforming? "
                 "Use search_reviews, then franchise_sales_trend, then compose_brief.")]

User picks /diagnose_franchise austin from a slash-menu. Server expands the templated workflow.

This is where opinionated workflows belong — not buried in tool docstrings.

Who picks what?

┌──────────────┬──────────────────────────────────────┐
│ Tool         │ Model picks. Model is in charge.     │
│ Resource     │ App / user picks. Indexable.         │
│ Prompt       │ User picks. Slash-command UX.        │
└──────────────┴──────────────────────────────────────┘

Most servers ship only tools. They miss half the spec.

The exam question: when you have a list of 50,000 entities, do you make a tool that returns the list, or a resource that the host can paginate? Answer: resource.

3. Beyond Tools

Three things people miss

  1. Sampling — server calls back to host's LLM
  2. Elicitation — server asks the user a question mid-call
  3. Notifications — server pushes updates to host

All three turn the host ↔ server conversation bidirectional.

Sampling — server uses host's LLM

@mcp.tool()
async def compose_brief(ctx: Context, franchise: str) -> str:
    reviews = search_reviews(franchise, q="complaint")
    trend   = franchise_sales_trend(franchise, days=90)

    # Server has NO API key. It asks the host's LLM to do the work.
    result = await ctx.session.create_message(
        messages=[{"role": "user", "content": f"Diagnose: {reviews}, {trend}"}],
        max_tokens=500,
    )
    return result.content.text

Why use it: no API key on the server, no model lock-in, host pays the bill, BYO-model from the host's perspective.

Elicitation — server asks the user

@mcp.tool()
async def log_finding(ctx: Context, franchise: str, hypothesis: str) -> str:
    response = await ctx.elicit(
        message="Confidence?",
        schema={"type": "string", "enum": ["low", "medium", "high"]},
    )
    confidence = response.content
    ...

Tool needs missing structured input? Don't fail or hallucinate — ask.

Host renders a picker UI. Tool resumes.

Notifications — server pushes updates

await ctx.session.send_resource_updated(uri="franchise://austin/recent_signals")

Resources can change. Server notifies, host re-fetches. No polling.

Use cases: live query results, evolving documents, long-running jobs that emit progress.

4. Transports & Auth

Two transports, one deprecated

TransportWhen
stdioLocal servers spawned by the host. Used by desktop hosts like Cursor / Claude Desktop.
Streamable HTTPRemote / hosted servers. Use for production. Resumable sessions.
SSEDeprecated. Streamable HTTP replaces it.

Section E hosts a FastMCP server on a Databricks App — that's Streamable HTTP behind workspace OAuth.

Auth — what the spec says

MCP defers auth to the transport. For Streamable HTTP, the spec defines OAuth 2.1 with dynamic client registration.

Production deployments use one of:

  • OAuth bearer tokens — server validates JWT per request.
  • Identity passthrough — host forwards the user's token; server queries downstream as that user. Gold standard.
  • Service principal — server holds its own credential. Easier, weaker attribution.

How Databricks handles it (in-workspace)

When you use Genie Code or AI Playground, the host is already authenticated as you. Tool calls hit managed MCP endpoints with your workspace identity.

Genie Code → managed Genie/VS/UC MCP → Databricks REST as YOU → UC enforces ACLs

No CLI login. No tokens to paste. The browser session is the credential.

Identity passthrough on Databricks Apps

If you host a FastMCP server on a Databricks App (covered as pattern, not built today):

@app.middleware("http")
async def capture_user(request, call_next):
    token = request.headers.get("x-forwarded-access-token")
    email = request.headers.get("x-forwarded-email")

UC functions are MCP tools

CREATE OR REPLACE FUNCTION main.workshop.franchise_health(franchise STRING)
RETURNS DOUBLE
LANGUAGE PYTHON AS $$ ... $$;

The Databricks UC Functions MCP server exposes every UC function in a schema as a tool, with permissions enforced by UC.

Rule of thumb: if it's "read or compute over Delta," make it a UC function. Reach for FastMCP only for sampling, elicitation, resources, or non-Delta side effects.

Genie is a tool, not the agent

your agent
  ├─ ask_genie:        "Crumbly Creations daily revenue, full window"
  ├─ query_vs_index:   "Austin reviews mentioning waiting"
  ├─ you.com search:   "Austin coffee competition May 2024"
  ├─ sampling:         "summarize all of the above"
  └─ log_finding:      writes to Delta

The mistake is making Genie itself the agent. Genie does one thing well: text-to-SQL on a curated space. Use it as a tool, then orchestrate around it.

Production patterns (appendix)

  • Gateway pattern — one server fronts many. Single auth, single observability point.
  • Identity passthrough — host's user identity ➞ Databricks Apps ➞ UC.
  • Audit + lineage — Delta tables for mcp_calls, mcp_lineage. AI/BI dashboard on top.
  • Resources for catalog UX — don't autocomplete 5K rows.

See docs/02-architecture.md for the full discussion.

What about Lakebase and Apps?

Free Edition supports Databricks Apps and Lakebase. We don't deploy in this workshop (keeping the agenda tight) — but the typical "MCP server in production on Databricks" stack is:

FastMCP server
  + Databricks App  (hosting + OAuth)
  + Lakebase        (low-latency state, e.g. findings, sessions)
